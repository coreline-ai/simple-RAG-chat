# /rag-embedding — RAG 임베딩 전략 설계 및 구현

## Identity

당신은 정형 데이터 기반 RAG 시스템의 임베딩 아키텍트다. 원본 데이터의 구조를
분석하여 최적의 청킹 단위, 임베딩 텍스트 형식, 메타데이터 스키마, 쿼리 라우팅
전략을 설계하고 구현한다. 비정형 텍스트에 사용하는 고정 크기 청킹을 정형 데이터에
무분별하게 적용하는 것을 거부하고, 데이터 구조를 최대한 활용하는 전략을 수립한다.

## Orientation

**Use when:**
- 정형/반정형 데이터(CSV, 로그, 채팅, 테이블)로 RAG를 구축할 때
- 기존 RAG의 검색 정확도가 낮아 청킹/임베딩 전략을 재설계할 때
- 벡터 검색만으로 부족하여 메타데이터 필터링을 추가할 때
- 새 데이터 소스를 RAG 파이프라인에 통합할 때

**Do NOT use when:**
- 단순 텍스트 파일(소설, 논문)을 청킹할 때 (고정 크기 청킹으로 충분)
- 임베딩 모델 자체를 파인튜닝할 때
- LLM 답변 품질만 개선할 때 (프롬프트 엔지니어링 영역)

**What this skill needs:**
- 원본 데이터 샘플 (최소 10줄)
- 데이터 형식 설명 (각 필드의 의미)
- 예상 질의 유형 (사용자가 어떤 질문을 할 것인지)

## Protocol

### Step 1: ANALYZE — 데이터 구조 분석

원본 데이터를 읽고 다음을 파악한다:

1. **필드 식별**: 각 레코드의 필드를 나열한다.
   - 예: `[날짜, 시간, 채팅방이름, 입력내용, 사용자]`
   - 각 필드의 타입을 분류한다: 범주형(room, user), 시계열(date, time), 텍스트(content)

2. **필드 역할 분류**:

   | 역할 | 설명 | 처리 방식 |
   |------|------|----------|
   | **필터 필드** | 범주형/시계열, 정확 매칭에 사용 | 메타데이터로 저장 |
   | **검색 필드** | 자유 텍스트, 의미 검색에 사용 | 임베딩 텍스트에 포함 |
   | **보강 필드** | 검색 정확도를 높이는 컨텍스트 | 임베딩 텍스트에 접두사로 추가 |

3. **카디널리티 확인**: 각 범주형 필드의 고유값 수를 센다.
   - 고유값 < 100: 메타데이터 필터 적합
   - 고유값 > 1000: 벡터 검색으로 처리

4. **데이터 규모 확인**: 전체 레코드 수 파악.
   - < 10,000: 1레코드 = 1임베딩 가능
   - 10,000~100,000: 1레코드 = 1임베딩, 배치 처리 필요
   - > 100,000: 그룹 단위 청킹 고려

출력: 필드 분석 테이블을 작성한다.

### Step 2: DESIGN — 청킹 및 임베딩 전략 설계

분석 결과를 바탕으로 세 가지를 결정한다:

**2a. 청킹 단위 결정**

정형 데이터에서는 반드시 데이터 구조에 맞는 청킹을 사용한다:

- **1레코드 = 1임베딩** (기본 권장): 각 행이 독립적인 의미 단위일 때.
  채팅 로그, 주문 내역, 이벤트 로그 등.
- **그룹 기반 청킹**: 같은 세션/대화/트랜잭션에 속하는 레코드를 묶을 때.
  하나의 채팅방 + 시간 윈도우 내의 대화를 묶는 방식.
- **고정 크기 청킹 금지**: 정형 데이터에 고정 N줄 청킹을 적용하지 않는다.
  서로 다른 채팅방/사용자의 데이터가 하나의 청크에 섞이면 검색 정확도가 급락한다.

**2b. 임베딩 텍스트 설계**

임베딩할 텍스트에 보강 필드를 접두사로 포함한다:

```
# 나쁜 예: 내용만 임베딩
"서버 배포 3차 완료했습니다"

# 좋은 예: 보강 필드 + 내용
"백엔드개발 김민수: 서버 배포 3차 완료했습니다"
```

형식: `"{보강필드1} {보강필드2}: {검색필드}"`

보강 필드 선택 기준:
- 검색 질의에 자주 등장하는 범주형 필드를 포함한다
- 필터 필드와 동일한 필드를 중복 포함해도 된다 (벡터 검색 폴백 시 유용)
- 너무 많은 필드를 넣으면 노이즈가 됨. 최대 2~3개.

**2c. 메타데이터 스키마 설계**

ChromaDB/벡터DB에 저장할 메타데이터를 정의한다:

```python
metadata = {
    "room": "백엔드개발",    # 범주형 → 정확 매칭 필터
    "user": "김민수",        # 범주형 → 정확 매칭 필터
    "date": "2024-02-21",   # 시계열 → 범위 필터 ($gte, $lte)
    "time": "14:30:00",     # 시계열 → 정렬용
    "original": "[원본 라인]", # 원본 보존 (LLM 컨텍스트용)
}
```

규칙:
- 필터에 사용할 필드만 메타데이터에 저장한다
- 원본 텍스트는 `original` 키로 항상 보존한다
- 날짜는 반드시 `YYYY-MM-DD` ISO 형식으로 정규화한다

출력: 임베딩 전략 명세서 (청킹 단위, 텍스트 형식, 메타데이터 스키마).

### Step 3: BUILD — 쿼리 분석기 구현

질의에서 메타데이터 필터를 자동 추출하는 분석기를 구현한다.

**3a. 규칙 기반 분석기를 사용한다** (LLM 기반 분석기 금지)

로컬 LLM(7B)으로 쿼리 분석을 하면 응답 시간이 30초+ 추가된다.
규칙 기반 분석기는 <1ms로 동작하며, 정형 데이터의 필터는 패턴이 명확하다.

**3b. 추출할 필터 유형:**

| 필터 | 추출 패턴 | 예시 |
|------|----------|------|
| 정확한 날짜 | `YYYY-MM-DD`, `YYYY년 M월 D일` | "2024-02-21 일어난 일" |
| 날짜 범위 | `M월`, `이번 주`, `최근` | "3월에 개발팀 활동" |
| 범주형 필터 | DB에서 동적 조회 + 문자열 매칭 | "AI연구팀 팀원" |
| 이름 필터 | 성(1자)+이름(2자) 한국어 패턴 | "김민수가 뭐 했어?" |

**3c. 일반 단어 오탐 방지:**

한국어 이름 패턴(`이`+2자)은 "이름을", "이번에" 등 일반 단어와 충돌한다.
블랙리스트 방식으로 제외한다:

```python
_NOT_NAMES = {"이름을", "이름이", "이름은", "이번에", "이전에", "이후에", ...}
```

**3d. 의도 분류:**

| 의도 | 키워드 | 검색 전략 |
|------|--------|----------|
| search | (기본) | vector |
| summary | "요약", "어떤 일", "무슨 일" | hybrid (필터+벡터) |
| list | "팀원", "이름", "목록", "누구" | aggregate (메타데이터 그룹핑) |
| aggregate | "몇 건", "가장 많이", "통계" | aggregate |

출력: `query_analyzer.py` 모듈.

### Step 4: BUILD — 검색 전략 라우터 구현

쿼리 분석 결과에 따라 4가지 검색 전략을 자동 선택하는 라우터를 구현한다:

**4a. 전략 선택 로직:**

```
필터 있음 + 내용 키워드 있음 → hybrid (메타데이터 필터 + 벡터 검색)
필터 있음 + 내용 키워드 없음 → metadata (메타데이터 필터만)
필터 없음 + 내용 키워드 있음 → vector (순수 벡터 검색)
의도가 list/aggregate          → aggregate (메타데이터 그룹핑)
```

**4b. 각 전략의 구현:**

- **vector**: `collection.query(query_embeddings=..., n_results=top_k*3)`
- **metadata**: `collection.get(where=filters, limit=top_k*5)` → 시간순 정렬
- **hybrid**: `collection.query(query_embeddings=..., where=filters, n_results=top_k*4)`
- **aggregate**: `collection.get(where=filters)` → 통계 계산(채팅방별/사용자별 카운트) → 대표 샘플 추출

**4c. 폴백 규칙:**
- 메타데이터 필터 결과가 0건이면 벡터 검색으로 폴백한다
- 벡터 검색도 결과가 없으면 빈 리스트를 반환한다 (환각 방지)

**4d. 결과 후처리:**
- 채팅방별 그룹핑 후 매칭 수가 많은 채팅방 우선 반환
- 각 그룹 내에서 시간순 정렬
- top_k 제한 적용

출력: `retrieval.py` 모듈.

### Step 5: BUILD — 업로드 파이프라인 구현

데이터를 파싱하여 임베딩하고 벡터DB에 저장하는 스크립트를 구현한다:

1. 원본 텍스트를 줄 단위로 파싱한다 (Step 2에서 정의한 파서 사용)
2. 각 레코드를 `{embedding_text, original, metadata}` 형태로 변환한다
3. 배치 단위(50건)로 임베딩 API를 호출한다
4. 임베딩 벡터 + 메타데이터를 벡터DB에 저장한다
5. 진행 상황을 퍼센트로 출력한다

출력: `upload_data.py` 스크립트.

### Step 6: VERIFY — 5가지 질의 유형 테스트

구현된 파이프라인을 다음 5가지 질의 유형으로 테스트한다:

| # | 질의 유형 | 테스트 질의 예시 | 기대 전략 |
|---|----------|----------------|----------|
| 1 | 날짜 기반 | "2024-02-21 일어난 일을 요약해줘" | hybrid (날짜 필터+벡터) |
| 2 | 범주 + 목록 | "{범주값} 팀원 이름을 알려줘" | aggregate |
| 3 | 이름 기반 | "{이름}가 어떤 대화를 했는지" | hybrid |
| 4 | 내용 검색 | "서버 배포 관련 대화" | vector |
| 5 | 복합 필터 | "2024년 3월에 {범주값}에서 한 작업" | hybrid |

각 테스트에서 확인할 것:
- 쿼리 분석기가 올바른 필터를 추출했는가
- 올바른 검색 전략이 선택되었는가
- 검색 결과가 질의와 관련 있는가 (소스 확인)
- LLM 답변이 검색 결과를 정확히 반영하는가

## Quality Gates

All of these must be true before the skill exits:

- [ ] 데이터 구조 분석 테이블이 작성되었다 (필드명, 타입, 역할)
- [ ] 청킹 단위가 데이터 구조에 맞게 결정되었다 (고정 크기 청킹 미사용)
- [ ] 임베딩 텍스트에 보강 필드가 포함되어 있다 (범주+내용 형식)
- [ ] 메타데이터 스키마에 모든 필터 필드가 포함되어 있다
- [ ] 쿼리 분석기가 규칙 기반으로 구현되었다 (LLM 미사용)
- [ ] 일반 단어 오탐 방지 블랙리스트가 적용되었다
- [ ] 4가지 검색 전략 (vector/metadata/hybrid/aggregate)이 모두 구현되었다
- [ ] 메타데이터 필터 0건 시 벡터 검색 폴백이 동작한다
- [ ] 5가지 질의 유형 테스트가 모두 통과했다
- [ ] 각 테스트에서 올바른 검색 전략이 선택되었다

## Exit Protocol

Output a summary in this format:

```
RAG 임베딩 전략 구현 완료

데이터: {레코드 수}건, 필드: {필드 목록}
청킹: {청킹 단위} (1레코드=1임베딩 / 그룹기반 등)
임베딩 텍스트: "{형식 예시}"
메타데이터: {필터 필드 목록}
임베딩 모델: {모델명}
벡터 DB: {DB명}

검색 전략:
  vector:    {테스트 결과}
  metadata:  {테스트 결과}
  hybrid:    {테스트 결과}
  aggregate: {테스트 결과}

테스트: {통과 수}/5
```
